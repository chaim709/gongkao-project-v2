# 公考培训机构管理系统 - 项目背景

## 一、业务背景

### 机构信息
- **名称**: 新途径教育（江苏）
- **业务**: 江苏公务员考试和事业编考试线下培训
- **规模**: 中小型培训机构
- **学员数**: 约200-500人/年
- **教师数**: 10-20人

### 核心痛点
1. **学员管理混乱** - Excel表格管理，效率低
2. **督学跟进不及时** - 依赖人工记忆，容易遗漏
3. **数据分析困难** - 无法快速了解学员学习情况
4. **选岗效率低** - 手动匹配岗位，耗时长
5. **家长沟通成本高** - 重复回答相同问题

## 二、核心功能需求

### 2.1 学员管理模块

**基本信息**：
- 姓名、电话、微信、身份证
- 学历、专业、政治面貌
- 户籍信息（省、市）
- 家长联系方式

**报考信息**：
- 报考类型（国考/省考/事业编）
- 目标岗位
- 考试日期
- 班次信息

**学习画像**：
- 是否有基础
- 基础水平（优秀/良好/一般/较差）
- 学习风格
- 个性化督学计划

**课程信息**：
- 报名套餐
- 课程报名日期
- 有效期
- 实付金额、优惠信息

**状态管理**：
- 缴费状态
- 学习状态
- 需重点关注标记
- 最后联系日期

### 2.2 督学管理模块

**督学日志**：
- 日期、时间
- 督学人员
- 沟通方式（电话/微信/面谈）
- 学员心态（积极/平稳/焦虑/低落）
- 学习状态（优秀/良好/一般/较差）
- 日志内容（支持常用短语快捷输入）
- 下次跟进时间

**跟进提醒**：
- 超过7天未跟进自动提醒
- 重点关注学员优先提醒
- 督学任务分配

**数据统计**：
- 督学工作量统计
- 学员跟进频率
- 心态变化趋势

### 2.3 课程管理模块

**课程信息**：
- 课程名称、类型
- 开课时间、结课时间
- 授课教师
- 课程内容

**作业管理**：
- 作业发布
- 作业提交
- 批改反馈
- 完成率统计

**打卡功能**：
- 每日打卡
- 连续打卡天数
- 打卡排行榜

### 2.4 智能选岗模块

**岗位信息**：
- 岗位名称、单位
- 专业要求
- 学历要求
- 政治面貌要求
- 基层工作年限
- 户籍限制

**智能匹配**：
- 基于学员信息自动匹配
- 多维度筛选
- 匹配度评分
- 推荐理由

### 2.5 数据分析模块

**学员统计**：
- 总学员数、新增学员
- 在读学员、已毕业学员
- 班次分布、报考类型分布

**学习分析**：
- 作业完成率
- 打卡率
- 学习进度

**督学分析**：
- 督学工作量
- 跟进及时率
- 学员满意度

**趋势分析**：
- 学员增长趋势
- 学习效果趋势
- 各项指标对比

### 2.6 学员端功能（未来扩展）

**学员登录**：
- 用户名/密码登录
- 查看个人信息

**学习管理**：
- 查看督学日志
- 查看学习进度
- 查看课程安排

**作业打卡**：
- 提交作业
- 查看作业批改
- 每日打卡
- 打卡统计

**智能选岗**：
- 查看岗位推荐
- 岗位匹配度分析
- 收藏岗位

## 三、技术要求与架构目标

### 3.1 技术栈选择

**后端**：
- **框架**: FastAPI（现代、高性能、自动API文档）
- **数据库**: PostgreSQL 14+
- **ORM**: SQLAlchemy 2.0
- **认证**: JWT + OAuth2
- **数据验证**: Pydantic
- **异步支持**: async/await

**前端**：
- **框架**: React 18 + TypeScript
- **UI库**: Ant Design Pro / shadcn/ui
- **状态管理**: Zustand / TanStack Query
- **构建工具**: Vite
- **HTTP客户端**: Axios

**部署**：
- **容器化**: Docker + docker-compose
- **反向代理**: Nginx
- **进程管理**: Gunicorn + Uvicorn

### 3.2 V2 版本改进目标
- ✅ 清晰的分层架构（API/Service/Repository/Model）
- ✅ 统一的异常处理和错误码体系
- ✅ 完善的日志系统（结构化日志）
- ✅ 软删除机制（保留数据历史）
- ✅ 操作审计日志（记录所有关键操作）
- ✅ 标准化的 API 设计（RESTful + 自动文档）
- ✅ 完善的权限控制（RBAC）
- ✅ 模块化设计（易于扩展）

### 3.3 性能要求
- 支持 200-500 并发用户
- API 响应时间 < 500ms（P95）
- 数据库查询优化（索引、分页）
- 支持水平扩展

### 3.4 安全要求
- JWT 认证 + RBAC 权限控制
- HTTPS 加密传输
- 敏感信息脱敏（手机号、身份证）
- 操作审计日志
- SQL 注入防护
- XSS/CSRF 防护

### 3.5 部署要求
- Docker 容器化部署
- 数据库定期备份（每日自动）
- 日志集中管理
- 健康检查接口
- 环境变量配置

## 四、现有系统分析

### 4.1 现有系统优点
- 功能完整，覆盖核心业务
- 数据模型设计合理
- 已有生产数据

### 4.2 现有系统问题
- SQLite数据库不稳定（路径问题）
- Flask框架较老
- 前端技术栈陈旧
- 缺少Docker部署
- 代码结构需优化

### 4.3 V2 版本改进目标
- ✅ PostgreSQL 替代 SQLite，提升稳定性
- ✅ FastAPI 替代 Flask，提升性能和开发体验
- ✅ React + TypeScript，提升前端可维护性
- ✅ Docker 容器化，简化部署
- ✅ 清晰的分层架构，提升代码质量
- ✅ 自动生成 API 文档，方便集成
- ✅ 模块化设计，易于扩展
- ✅ 移除小程序，专注核心系统

### 4.4 数据迁移策略
V2 版本将从零开始，不迁移旧系统数据。理由：
- 旧系统数据结构需要大幅调整
- 新系统添加了软删除、审计日志等新字段
- 避免数据迁移风险，确保新系统稳定性
- 旧系统可保留作为历史参考
